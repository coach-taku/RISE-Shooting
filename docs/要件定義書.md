# AIアプリ開発 要件定義書

本要件定義書は、バスケットボールの選手個人のシューティングデータを蓄積・分析し、チーム内のモチベーション向上と自己管理能力の育成を目的とした、Week2プレビューアプリ用の要件定義書である。AIエージェント（GenSpark）での開発を前提とし、最小限かつ効果的な構成とする。

> **改訂履歴**
> - v2（2026-05-17）: ログイン方式をメールアドレス入力 → 名前プルダウン選択に変更（デモ版対応）
> - v3（2026-05-20）: セキュリティ強化 — 共通の「合言葉」から個別パスワード管理へ移行。アカウント管理画面にパスワード一覧表示・名前/パスワード編集機能を追加。
> - v4（2026-05-21）: ログイン状態の永続化対応 — アプリを閉じて再起動してもログイン状態を維持する機能を追加。
> - v4.1（2026-05-22）: middleware.ts を新規作成し Next.js Middleware としてセッション自動リフレッシュを正式に有効化。proxy.ts は廃止（Next.js は "middleware.ts" という固定ファイル名のみを Middleware として認識するため、proxy.ts では実際に実行されていなかった不具合を修正）。
> - v4.2（2026-05-22）: Next.js 16 対応 — Next.js 16 では "middleware.ts" が非推奨となり "proxy.ts"（関数名: `proxy`）が正式な規約ファイル名に変更された。これに伴い middleware.ts を削除し proxy.ts へ移行。ログイン後に `router.refresh()` を追加し、サーバー側 Cookie とクライアント状態の同期を確実化。
> - v5（2026-05-22）: フリースロー（FT）エリアを追加 — シュート記録入力・個人集計・コーチダッシュボードのエリア選択肢に「フリースロー（FT）」を追加。既存データへの影響なし（area_name の文字列追加のみ、DBマイグレーション不要）。
> - v6（2026-05-23）: カレンダー日付選択によるデータ入力機能の改善 — シュート記録入力画面（P-02）にインライン表示型カレンダーUIを追加。過去・未来を問わず任意の日付を選択して記録できるよう改修。選択日の既存データを自動読み込みしてフォームに反映する機能も追加。DBスキーマ変更なし。
> - v7（2026-05-23）: シューティング記録画面のバグ修正 — ①入力セットを最大10セットに制限（従来の最大5セットから拡張）。②各セットに一意IDを付与し入力値の連動バグを解消（1つの入力が他セットに波及しない独立した状態管理を実現）。③送信時に空セットをフィルタリングして未入力行をDBに保存しないよう修正。④Service Worker（sw.js）のエラーハンドリングを改善しFetchEventのネットワークエラー時に必ずResponseを返すよう修正。
> - v8（2026-05-23）: 過去データの修正・削除機能を追加 — バグにより過去の入力データに誤りが発生した場合に備え、選手が自身の過去データを遡って修正・削除できる機能をシュート記録入力画面（P-02）に追加。①カレンダーで選択した日付に既存データがある場合「修正モード」に切り替わり、「修正・保存」ボタンと「削除」ボタンが表示される。②修正は既存レコードを一括削除してから新データを再INSERT する方式で実現（DBスキーマ変更なし）。③削除時は確認モーダルを表示し誤操作を防止。④RLSポリシー確認済み（records_delete_own・records_insert_own が既存）、将来のUPDATE方式切替用SQLを supabase/add-edit-delete-policy.sql に追加。

## 1. 機能要件

### 1.1. 共通機能
- **認証機能**
  - ログイン画面では「名前（プルダウン選択）」と「パスワード」の組み合わせで認証する。
  - メールアドレスの入力は不要。あらかじめ登録されたユーザー名からプルダウンで選択する形式とする。
  - 内部では選択した名前に対応する仮想メールアドレス（例: `yamada@rise-shooting.example.com`）を自動的に紐付け、Supabase Auth のメール認証機能を裏側で再利用する。
  - 各ユーザーは固有のパスワードを持つ（共通の合言葉は廃止）。
  - **ログイン状態の永続化**：一度ログインに成功した後は、明示的に「ログアウト」しない限り、アプリを閉じて再起動してもログイン済みの状態から開始される。セッションは Cookie に永続保存（有効期限: 最大400日）され、サーバー側（Next.js Proxy）で自動リフレッシュされる。
  - **デモ用初期アカウント（シードデータ）**
    - コーチ山田（コーチ権限） / 初期パスワード: rise2024
    - 選手佐藤（選手権限） / 初期パスワード: rise2024
    - 選手鈴木（選手権限） / 初期パスワード: rise2024
    - ※ 初期パスワードはコーチが管理画面からいつでも変更可能

### 1.2. 選手向け機能
- **シューティング記録入力画面**
  - 練習直後に体育館で、スマホから30秒程度で入力完了できるUI/UX。
  - 1回の入力で「最大10エリア分」をまとめて登録できるフォーム構造とする（v7改善: 従来の5エリアから10エリアに拡張）。
  - 各セット（エリア入力枠）は独立した入力状態を持ち、1つの枠への入力が他の枠に波及・連動しない（v7修正）。
  - 送信時、エリアと成功数の両方が入力されているセットのみDBに保存する。空の枠は送信しない（v7修正）。
  - **インラインカレンダーによる日付選択（v6追加）**：
    - 記録入力画面の上部に、ポップアップではなく常時表示されるカレンダーUIを配置する。
    - 過去・現在・未来のいずれの日付も制限なく選択できる（未来日付の選択禁止なし）。
    - カレンダーで日付を選択した際、その日に既存データがあれば自動的にフォームへ読み込み、追記・確認が可能。
    - 保存時は選択した日付のデータとしてDBに登録（insert）される。既存データの意図しない削除・上書きは行わない。
  - **過去データの修正・削除機能（v8追加）**：
    - カレンダーで選択した日付に既存データがある場合、画面が「修正モード」に切り替わる。
    - 修正モードでは「修正・保存」ボタンと「削除」ボタンが表示される（通常の「登録」ボタンは非表示になる）。
    - **修正（上書き更新）**：フォームの内容を編集して「修正・保存」ボタンを押すと、その日の既存データが削除され、編集後のデータが新たに登録される。
    - **削除**：「削除」ボタンを押すと確認モーダルが表示され、「削除する」を選ぶとその日のデータがすべて削除される。誤操作防止のため確認ステップを必ず挟む。
    - 削除後はフォームがリセットされ、新規入力状態に戻る。
    - 遡れる期間の制限はなし（カレンダーで選択可能なすべての日付が対象）。
  - 各エリアの入力項目：
    - **エリア選択**：プルダウンメニューから選択
      - 3ポイントエリア（5分割）
        - 左コーナー
        - 左ウイング
        - トップ
        - 右ウイング
        - 右コーナー
      - 2ポイントエリア：ミドルレンジ（5分割）
        - 左ショートコーナー
        - 左ミドル
        - トップミドル
        - 右ミドル
        - 右ショートコーナー
      - 2ポイントエリア：インサイド（2分割）
        - ペイントエリア中央
        - ゴール下
      - **フリースロー（FT）**（v5追加）
        - フリースロー（FT）
    - **シュート本数（試投数）**：試投数は入力せず、試投数10本ごとに成功数を入力するのみ
    - **成功数（入った数）**：1〜10の範囲で選択または入力。
  - 登録完了時に、選手のモチベーションを高め「今を全力で愉しむ」ことを後押しするポジティブな応援メッセージを表示する。

- **個人データ閲覧・期間集計画面**
  - **本日の記録**：その日に登録したシュート結果（エリアごとの本数、成功数、確率）を即座にフィードバック。
  - **期間集計機能**：ユーザーがカレンダー等で任意の期間（週・月・指定期間）を選択し、その期間内のエリアごとの総試投数、総成功数、平均成功確率を集計して閲覧できる。
  - **サマリーカード**：FG%・3P%・2P%・**FT%**（v5追加）のカテゴリ別成績を表示。ショットチャートの下にはFT専用スタッツカードを表示する。

### 1.3. コーチ（管理者）向け機能
- **管理・分析ダッシュボード画面（PC・タブレット閲覧推奨）**
  - チーム全体のシューティング状況を把握し、選手への声かけをサポートする画面。
  - **努力量ラインキング**：総シュート試投数が多い「たくさん打って努力している選手」を一覧で把握できる。
  - **エリア別トップスタッツ**：エリアごとに、最も高い成功確率を記録している選手を一覧表示する。
- **アカウント管理機能（セキュリティ強化版）**
  - 選手の一覧と各選手に設定された現在のパスワードを確認できる（一括表示/非表示の切り替えあり）。
  - コーチが選手の名前（表示名）を管理画面から直接変更できる。
  - コーチが選手のパスワードを管理画面から直接変更できる（変更は Supabase Auth に即時反映）。
  - 変更したパスワードはデータベースにも保存され、Supabase ダッシュボードから直接確認・更新することも可能。

## 2. 非機能要件

### 2.1. 開発・ホスティング環境
- **開発方式**：AIエージェント（GenSpark）による自動コーディングにて実施。人間による直接のコード編集は行わない。
- **データベース**：Supabase（PostgreSQL）を使用。
- **ホスティング**：Vercel上の環境にホスティングして運用。

### 2.2. 対応デバイス・UI/UX
- **モバイルファースト**：選手が練習直後の体育館でスムーズに入力できるよう、入力画面および個人閲覧画面はスマートフォンでの操作・閲覧に完全最適化する。
- **レスポンシブ対応**：コーチがPCやタブレットで大画面チェックできるよう、管理画面はレスポンシブ対応とする。
- **UIカラーテーマ**：
  - 背景色：`#c0c0bf`（落ち着いたグレー系）
  - メインカラー：`#e1c614`（アクティブなイエロー/ゴールド系）
  - アクセントカラー：`#ffffff`（クリーンなホワイト）
- **トーン＆マナー**：高校生選手が強制感を感じず、自発的に入力を愉しめるよう、ポジティブで親しみやすい言葉づかいのUI文言を採用する。

### 2.3. スコープアウト（あえて実装しないこと）
- 動画や画像のアップロード機能（アプリの軽量化・シンプル運用の維持のため）。
- 複雑なチャットやダイレクトメッセージ機能（指導者からの声かけはリアルで行う想定のため）。
- メール通知やプッシュ通知機能。

## 4. データ構造（データモデル）

### 4.1. Users（ユーザー情報）
選手およびコーチのアカウント情報を管理する。

| フィールド名 | データ型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| user_id | UUID (PK) | ユーザー固有の識別子 | Supabase Auth の auth.users.id と連携 |
| username | 文字列 | 選手名またはコーチ名 | ログインに使用 |
| password_plain | 文字列 | パスワード（平文・表示用） | コーチが管理画面で確認できる。実際の認証は Supabase Auth が担当 |
| role | 文字列 | 権限（'player' = 選手, 'coach' = コーチ） | 画面の切り替えに使用 |
| created_at | タイムスタンプ | アカウント作成日時 | 自動設定 |

### 4.2. ShootingRecords（シューティング記録）
選手が入力した日々のシュートデータを蓄積する。

| フィールド名 | データ型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| record_id | 整数 (PK) | 記録固有の識別子 | 自動採番 |
| user_id | UUID (FK) | 入力した選手のユーザーID | Usersテーブルとリレーション |
| date | 日付 | シュートを実施・記録した日付 | 期間集計の基準 |
| area_name | 文字列 | シュートを打ったエリア名 | プルダウン選択値 |
| attempts | 整数 | 試投数（シュートを打った本数） | 10本単位 |
| successes | 整数 | 成功数（シュートが決まった本数） | 1〜10の範囲 |

## 5. セキュリティ設計（v3追加・v4更新）

### 5.1. 認証フロー
1. ログイン画面でユーザーが「名前（プルダウン）」と「パスワード」を入力
2. サーバーAPIが profiles テーブルから名前に対応するユーザーIDを取得
3. auth.users からダミーメールアドレスを逆引き
4. Supabase Auth の `signInWithPassword` で認証（パスワード照合は Supabase Auth が担当）
5. 認証成功後、セッション（JWT）が Cookie に永続保存される（有効期限: 最大400日）
6. role に応じて選手画面 or コーチ画面にリダイレクト
7. 以降のアクセス時は Next.js Proxy（`proxy.ts`）がセッション Cookie を自動検証・リフレッシュし、ログイン状態を維持する

### 5.2. ログイン状態の永続化（v4追加・v4.2更新）
- **セッションの保持方法**: Supabase Auth のセッション（JWTトークン）をブラウザの Cookie に永続保存する（`maxAge: 400日`）。
- **自動リフレッシュ**: Next.js 16 の Proxy（`proxy.ts`）がすべてのリクエストでセッション Cookie を検証・リフレッシュする。JWT の有効期限（デフォルト1時間）が切れても自動更新されるため、ログアウト操作をしない限りログイン状態が継続する。
- **ログイン後のセッション同期**: ログイン成功時に `router.refresh()` を呼び出し、サーバー側が発行した Cookie をクライアント（Server Components）に即時同期する。
- **ログアウト**: 明示的なログアウト操作（NavBar のログアウトボタン）を行うことで、Cookie が削除されログイン状態が解除される。

### 5.3. パスワード管理
- **パスワードの変更**: コーチがアカウント管理画面（M-02）から変更 → Supabase Auth Admin API で即時反映 → `profiles.password_plain` にも同期保存
- **Supabaseダッシュボードからの直接変更**: `profiles.password_plain` を書き換えても、Supabase Auth 側は自動同期されない。必ずアプリの管理画面から変更するか、Admin API 経由でパスワードを更新すること。
- **平文保存について**: `password_plain` は管理者（コーチ）が選手のパスワードを把握するための運用上の利便性を優先した設計。より高いセキュリティが必要になった場合は、この仕組みの廃止を検討する。
