# AIアプリ開発 要件定義書

本要件定義書は、バスケットボールの選手個人のシューティングデータを蓄積・分析し、チーム内のモチベーション向上と自己管理能力の育成を目的とした、Week2プレビューアプリ用の要件定義書である。AIエージェント（GenSpark）での開発を前提とし、最小限かつ効果的な構成とする。

> **改訂履歴**
> - v2（2026-05-17）: ログイン方式をメールアドレス入力 → 名前プルダウン選択に変更（デモ版対応）
> - v3（2026-05-20）: セキュリティ強化 — 共通の「合言葉」から個別パスワード管理へ移行。アカウント管理画面にパスワード一覧表示・名前/パスワード編集機能を追加。
> - v4（2026-05-20）: パスワード最小文字数を6文字以上 → **4文字以上**に変更（選手のパスワード忘れ防止のため）。
> - v5（2026-05-20）: パスワードパディング方式を導入。Supabase Auth の6文字制限を回避しつつ4文字パスワードを実現。

---

## 1. 機能要件

### 1.1. 共通機能
- **認証機能**
  - ログイン画面では「名前（プルダウン選択）」と「パスワード」の組み合わせで認証する。
  - メールアドレスの入力は不要。あらかじめ登録されたユーザー名からプルダウンで選択する形式とする。
  - 内部では選択した名前に対応する仮想メールアドレス（例: `yamada@rise-shooting.example.com`）を自動的に紐付け、Supabase Auth のメール認証機能を裏側で再利用する。
  - 各ユーザーは固有のパスワードを持つ（共通の合言葉は廃止）。
  - **デモ用初期アカウント（シードデータ）**
    - コーチ山田（コーチ権限） / 初期パスワード: rise2024
    - 選手佐藤（選手権限） / 初期パスワード: rise2024
    - 選手鈴木（選手権限） / 初期パスワード: rise2024
    - ※ 初期パスワードはコーチが管理画面からいつでも変更可能

### 1.2. 選手向け機能
- **シューティング記録入力画面**
  - 練習直後に体育館で、スマホから30秒程度で入力完了できるUI/UX。
  - 1回の入力で「5つのエリア分」をまとめて登録できるフォーム構造とする。
  - 各エリアの入力項目：
    - **エリア選択**：プルダウンメニューから選択
      - 3ポイントエリア（5分割）
        - 左コーナー
        - 左ウイング
        - トップ
        - 右ウイング
        - 右コーナー
      - 2ポイントエリア：ミドルレンジ（5分割）
        - 左ショートコーナー
        - 左ミドル
        - トップミドル
        - 右ミドル
        - 右ショートコーナー
      - 2ポイントエリア：インサイド（2分割）
        - ペイントエリア中央
        - ゴール下
    - **シュート本数（試投数）**：試投数は入力せず、試投数10本ごとに成功数を入力するのみ
    - **成功数（入った数）**：1〜10の範囲で選択または入力。
  - 登録完了時に、選手のモチベーションを高め「今を全力で愉しむ」ことを後押しするポジティブな応援メッセージを表示する。

- **個人データ閲覧・期間集計画面**
  - **本日の記録**：その日に登録したシュート結果（エリアごとの本数、成功数、確率）を即座にフィードバック。
  - **期間集計機能**：ユーザーがカレンダー等で任意の期間（週・月・指定期間）を選択し、その期間内のエリアごとの総試投数、総成功数、平均成功確率を集計して閲覧できる。

### 1.3. コーチ（管理者）向け機能
- **管理・分析ダッシュボード画面（PC・タブレット閲覧推奨）**
  - チーム全体のシューティング状況を把握し、選手への声かけをサポートする画面。
  - **努力量ラインキング**：総シュート試投数が多い「たくさん打って努力している選手」を一覧で把握できる。
  - **エリア別トップスタッツ**：エリアごとに、最も高い成功確率を記録している選手を一覧表示する。
- **アカウント管理機能（セキュリティ強化版）**
  - 選手の一覧と各選手に設定された現在のパスワードを確認できる（一括表示/非表示の切り替えあり）。
  - コーチが選手の名前（表示名）を管理画面から直接変更できる。
  - コーチが選手のパスワードを管理画面から直接変更できる（変更は Supabase Auth に即時反映）。
  - 変更したパスワードはデータベースにも保存され、Supabase ダッシュボードから直接確認・更新することも可能。
  - **パスワードは4文字以上で設定可能**（記憶しやすさを優先）。

## 2. 非機能要件

### 2.1. 開発・ホスティング環境
- **開発方式**：AIエージェント（GenSpark）による自動コーディングにて実施。人間による直接のコード編集は行わない。
- **データベース**：Supabase（PostgreSQL）を使用。
- **ホスティング**：Vercel上の環境にホスティングして運用。

### 2.2. 対応デバイス・UI/UX
- **モバイルファースト**：選手が練習直後の体育館でスムーズに入力できるよう、入力画面および個人閲覧画面はスマートフォンでの操作・閲覧に完全最適化する。
- **レスポンシブ対応**：コーチがPCやタブレットで大画面チェックできるよう、管理画面はレスポンシブ対応とする。
- **UIカラーテーマ**：
  - 背景色：`#c0c0bf`（落ち着いたグレー系）
  - メインカラー：`#e1c614`（アクティブなイエロー/ゴールド系）
  - アクセントカラー：`#ffffff`（クリーンなホワイト）
- **トーン＆マナー**：高校生選手が強制感を感じず、自発的に入力を愉しめるよう、ポジティブで親しみやすい言葉づかいのUI文言を採用する。

### 2.3. スコープアウト（あえて実装しないこと）
- 動画や画像のアップロード機能（アプリの軽量化・シンプル運用の維持のため）。
- 複雑なチャットやダイレクトメッセージ機能（指導者からの声かけはリアルで行う想定のため）。
- メール通知やプッシュ通知機能。

## 4. データ構造（データモデル）

### 4.1. Users（ユーザー情報）
選手およびコーチのアカウント情報を管理する。

| フィールド名 | データ型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| user_id | UUID (PK) | ユーザー固有の識別子 | Supabase Auth の auth.users.id と連携 |
| username | 文字列 | 選手名またはコーチ名 | ログインに使用 |
| password_plain | 文字列 | パスワード（平文・表示用） | コーチが管理画面で確認できる。実際の認証は Supabase Auth が担当 |
| role | 文字列 | 権限（'player' = 選手, 'coach' = コーチ） | 画面の切り替えに使用 |
| created_at | タイムスタンプ | アカウント作成日時 | 自動設定 |

### 4.2. ShootingRecords（シューティング記録）
選手が入力した日々のシュートデータを蓄積する。

| フィールド名 | データ型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| record_id | 整数 (PK) | 記録固有の識別子 | 自動採番 |
| user_id | UUID (FK) | 入力した選手のユーザーID | Usersテーブルとリレーション |
| date | 日付 | シュートを実施・記録した日付 | 期間集計の基準 |
| area_name | 文字列 | シュートを打ったエリア名 | プルダウン選択値 |
| attempts | 整数 | 試投数（シュートを打った本数） | 10本単位 |
| successes | 整数 | 成功数（シュートが決まった本数） | 1〜10の範囲 |

## 5. セキュリティ設計（v3追加）

### 5.1. 認証フロー
1. ログイン画面でユーザーが「名前（プルダウン）」と「パスワード」を入力
2. サーバーAPIが profiles テーブルから名前に対応するユーザーIDを取得
3. auth.users からダミーメールアドレスを逆引き
4. Supabase Auth の `signInWithPassword` で認証（パスワード照合は Supabase Auth が担当）
5. 認証成功後、role に応じて選手画面 or コーチ画面にリダイレクト

### 5.2. パスワード管理

#### パスワード最小文字数と Supabase 制約の回避策（パディング方式）
- **ユーザーが設定するパスワード**: **4文字以上**（フロントエンド・バックエンドAPI共通）
  - 記憶しやすさを優先して6文字から緩和。
- **Supabase Auth の制約**: ダッシュボード上で「Minimum password length」は6未満に設定できない（「Must be greater or equal to 6」エラーが発生する）。
- **回避策（パディング方式）**: アプリ内部でユーザー入力パスワードに固定サフィックス `__rise` を付加してから Supabase Auth に渡す。

  ```
  ユーザーが入力  →  Supabase Auth に渡す値
  "1234"         →  "1234__rise"  （10文字 ≥ 6文字 ✓）
  "abcd"         →  "abcd__rise"  （10文字 ≥ 6文字 ✓）
  ```

  - **実装箇所**: `lib/constants.ts` の `buildAuthPassword()` 関数が一元管理する。
    ```typescript
    export const PASSWORD_SUFFIX = '__rise'
    export function buildAuthPassword(userPassword: string): string {
      return userPassword + PASSWORD_SUFFIX
    }
    ```
  - **適用箇所**（全4箇所）:
    1. `app/login/page.tsx` — ログイン時の `signInWithPassword`
    2. `app/api/admin/create-player/route.ts` — 選手作成時の `admin.createUser`
    3. `app/api/admin/reset-password/route.ts` — パスワードリセット時の `admin.updateUserById`
    4. `app/api/admin/update-player/route.ts` — パスワード更新時の `admin.updateUserById`
  - ⚠️ **`PASSWORD_SUFFIX` を変更すると既存ユーザー全員がログインできなくなるため、絶対に変更しないこと。**

#### パスワードの変更・表示
- **変更方法**: コーチがアカウント管理画面（M-02）から変更 → Supabase Auth Admin API で即時反映 → `profiles.password_plain` にも同期保存
- **`profiles.password_plain` に保存する値**: サフィックスを除いたユーザー入力値のみ（コーチが管理画面で確認するときに意図した文字列が表示されるようにするため）。
- **Supabaseダッシュボードからの直接変更禁止**: `profiles.password_plain` を直接書き換えても Supabase Auth 側は自動同期されない。必ずアプリの管理画面から変更するか、Admin API 経由でパスワードを更新すること。
- **平文保存について**: `password_plain` は管理者（コーチ）が選手のパスワードを把握するための運用上の利便性を優先した設計。より高いセキュリティが必要になった場合は、この仕組みの廃止を検討する。

#### 既存ユーザーのパスワード移行
- パディング方式導入前に登録されたユーザー（例: `rise2024` で登録）は、Supabase Auth 上では `rise2024` のまま保存されている。
- 導入後はログイン時に `rise2024__rise` が送信されるため、**既存ユーザーはコーチが管理画面から一度パスワードを再設定する必要がある**。
- 再設定後は `profiles.password_plain` の値（例: `rise2024`）を参照してパスワードを確認できる。
